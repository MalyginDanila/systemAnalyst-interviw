Предназначен для создания нового ресурса вида объект

* Свойства: Не идемпотентный, не безопасный(т.к влияет на сервер и изменяет информацию), не кэшируемый
* Запрос: POST api.com/users (т.е мы можем обращаться напрямую к коллекциям, хранилищам и тд без указания идентификатора) + тело запроса(со всей необходимой информацией) 
* Ответ: 200ок + заголовок(location: api.com/users/321) + тело ответа(содержит всю информацию о пользователе)
Создаётся новый объект у которого будет новый, назначенный сервером адрес

Для создания искусственной идемпотентности добавляется условно super_id в тело запроса(который генерируется клиентом один раз), который идентифицирует сущность вашего запроса. Таким образом сервер проверяет создавалась ли уже сущность с данным super_id. Т.е на все остальные попытки сервер ответит 208Ок, так как он уже делал эту операцию.

- Тело запроса должно содержать данные, по которым создается новый ресурс. В итоге создается новый ресурс - как будто новый файл. Сервер сам назначает ресурсу ID.
- В теле ответа обычно возвращают либо весь созданный ресурс, включая метаданные об ID ресурса и т. д., либо только метаданные об ID ресурса и т. д..
- Не забывайте про заголовок Location в ответе 
- С помощью отдельного ключа (SuperID) мы можем искусственно сделать POST идемпотентным (как и PATCH)
- Если используем ресурс вида контроллер, просто вызываем POST /controller. В теле запроса передаём данные клиента или всё, что необходимо для выполнения действия над другим ресурсом. Например, мы хотим чтобы контроллер отправил смс ресурсу sms/123 — пользователю 123
